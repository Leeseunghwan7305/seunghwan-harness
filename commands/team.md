승환님이 맡긴 작업: **$ARGUMENTS**

이 작업을 역할별 팀에게 넘겨 **계획**을 완성해줘. **실제 코드 수정이나 커밋은 하지 마.** 계획을 보여주고 멈춘 뒤, 승환님이 승인하면 그때 구현에 들어간다.

작업 설명이 비어 있으면(`$ARGUMENTS` 가 없으면), 무엇을 시킬지 먼저 물어보고 멈춰줘.

## 먼저: 역할 고르기

10명을 매번 다 부르지 마라. **작업에 실제로 필요한 역할만** 골라라. 진짜 팀도 티켓마다 전원 소집하지 않는다.

- `$ARGUMENTS` 가 `--all` 로 시작하면 → 관련된 모든 역할을 부른다.
- 그 외에는 → 작업 성격을 보고 필요한 역할만 고른다.

**누구를 부르고 누구를 왜 뺐는지 맨 처음에 한 줄로 밝혀라.** 조용히 건너뛰지 마라. (예: "이번 작업은 문서 한 줄 추가라 PM·문서만 부르고 나머지는 뺐습니다.")

역할 판단 기준:
- **PM** — 거의 항상. 요구사항이 애매하면 필수.
- **리서처(researcher)** — 모르는 라이브러리·외부 API·모범 사례 조사가 필요할 때.
- **아키텍트(architect)** — 구조 결정이 필요한 규모일 때. 사소한 변경엔 생략.
- **프론트엔드(frontend)** — UI·클라이언트를 건드릴 때.
- **백엔드(backend)** — 서버·데이터·API·일반 로직·CLI·스크립트를 건드릴 때.
- **코드 리뷰어(code-reviewer)** — 계획이 나오면 거의 항상.
- **QA(qa)** — 테스트 전략·엣지 케이스가 중요할 때.
- **성능(perf)** — 뜨거운 경로·대량 데이터·확장성이 걸릴 때.
- **보안(security)** — 인증·사용자 입력·비밀정보·데이터를 다룰 때.
- **문서(docs)** — 사용자에게 보이는 변경이나 공개 API가 바뀔 때.

## 진행 순서 (단계)

각 단계는 반드시 해당 **에이전트**에게 위임하고, 앞 단계 결과를 다음 단계로 넘겨라. 직접 다 하지 말고 역할을 실제로 나눠라 — 그게 이 커맨드의 핵심이다. **한 단계 안의 여러 역할은 병렬로 불러도 된다** (예: 프론트+백엔드 동시, 검증 4종 동시).

1. **이해** — `seunghwan-harness:pm` (요구사항). 필요하면 `seunghwan-harness:researcher` 도 함께.
2. **설계** — `seunghwan-harness:architect` (구조). 사소하면 생략.
3. **구현 계획** — `seunghwan-harness:frontend` / `seunghwan-harness:backend` 중 해당하는 쪽(들). 앞 단계 산출물을 넘긴다.
4. **검증** — `seunghwan-harness:code-reviewer` / `seunghwan-harness:qa` / `seunghwan-harness:perf` / `seunghwan-harness:security` 중 해당하는 쪽(들)에게 구현 계획을 넘겨 점검.
5. **문서** — `seunghwan-harness:docs` (필요할 때).

## 결과 정리

부른 역할들의 결과를 단계 순으로 합쳐서 보여줘. 특히:
- **요구사항** (PM)
- **구현 계획** (개발자) — 작업 순서 강조
- **검증 요약** — 각 리뷰어의 "막아야 함" 위주로

리뷰어가 "막아야 함" / "위험" 을 냈으면, 그걸 반영해 계획을 어떻게 고칠지 짧게 덧붙여줘.

## 멈추기
