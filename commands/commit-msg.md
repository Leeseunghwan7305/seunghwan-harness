승환님을 위해 커밋 메시지를 만들어줘. 커밋을 실행하지는 말고, 메시지 초안만 보여줘.

## 순서

1. `git diff --staged` 로 staged된 변경을 확인해줘.
2. staged된 게 없으면, `git status` 를 보여주고 "먼저 `git add` 로 파일을 stage 해주세요" 라고 알려준 뒤 멈춰줘.
3. `git log --format='%B' -3` 으로 이 저장소의 커밋 스타일을 확인해서 맞춰줘.

## 메시지 형식 (Conventional Commits)

- **첫 줄**: `<type>: <한글 요약>` 형태. 요약은 무엇을 왜 했는지 50자 이내, 마침표 없이.
- 한 줄 띄우고
- **본문**: 변경이 여러 개면 `- ` 불릿으로. 하나뿐이면 짧은 문장 한두 개로. "무엇을" 보다 "왜" 에 무게를 둬줘.
- 사소한 변경(오타, 포맷팅)이면 첫 줄만으로 충분해.

**type 은 변경 성격에 맞게 골라줘:**

| type | 언제 |
|---|---|
| `feat` | 새 기능 추가 |
| `fix` | 버그 수정 |
| `hotfix` | 운영/배포된 코드의 급한 긴급 수정 (일반 `fix` 보다 시급) |
| `refactor` | 동작은 그대로, 코드 구조만 개선 |
| `docs` | 문서·README·주석만 변경 |
| `test` | 테스트 추가·수정 |
| `chore` | 빌드·설정·의존성 등 잡일 |
| `style` | 포맷팅·공백 등 동작 무관한 변경 |
| `perf` | 성능 개선 |

한 커밋에 여러 성격이 섞였으면 **가장 핵심인 것** 하나로 골라줘 (그럴 땐 커밋을 나누는 게 나은지도 짚어줘). 범위를 밝히고 싶으면 `feat(team): ...` 처럼 괄호로 스코프를 넣어도 좋아.

`$ARGUMENTS` 에 방향 지시가 있으면 (예: "간단하게", "영어로") 그걸 따라줘. 없으면 위 기본값대로.

## 출력

메시지를 ```` ```text ```` 코드블록으로 감싸서 그대로 복사할 수 있게 보여줘.
그 아래에 바로 커밋할 수 있는 명령을 한 줄 덧붙여줘:

`git commit -m "..."` 형태로, 방금 만든 메시지를 넣어서.

승환님이 "커밋해줘" 라고 하기 전에는 **직접 커밋하지 마.**
