승환님이 맡긴 작업: **$ARGUMENTS**

개발자와 리뷰어를 **핑퐁**시켜라 — 개발자가 계획을 내면 리뷰어가 깐다. 리뷰어가 통과시킬 때까지 개발자가 고치고 리뷰어가 다시 본다. **리뷰어 OK 가 나면 종료.**

이 단계에서는 **코드를 짜지 않고 계획만** 다듬는다. (개발자·리뷰어 모두 읽기 도구만 있어 파일을 못 고친다.) 최종 승인된 계획을 보여주고 멈춘 뒤, 승환님이 승인하면 그때 구현한다.

작업 설명이 비어 있으면 무엇을 시킬지 먼저 물어보고 멈춰라.

## 핑퐁 규칙

- **최대 라운드**: 기본 3라운드. `$ARGUMENTS` 에 "5라운드" 같은 지시가 있으면 그 숫자로.
- **같은 개발자에게 이어서 넘겨라.** 매 라운드 새 개발자를 부르지 말고, **직전 라운드의 개발자에게 리뷰어 피드백을 전달**해서 자기가 짠 계획을 기억한 채로 고치게 하라. (그래야 진짜 핑퐁이다.)
- 리뷰어도 매 라운드 **직전에 본 계획과 뭐가 달라졌는지** 알 수 있게, 이전 지적을 함께 전달하라.

## 진행

**라운드 1**
1. 개발자를 부른다 — 작업이 UI·클라이언트면 `seunghwan-harness:frontend`, 서버·데이터·로직·CLI·문서 등이면 `seunghwan-harness:backend`. (둘 다 걸치면 필요한 쪽을 고르거나 둘 다.) 이 개발자에게 작업의 구현 계획을 세우게 한다.
2. `seunghwan-harness:code-reviewer` 에게 그 계획을 넘겨 점검시킨다.

**판정**
- 리뷰어 결과에 **"막아야 함"** 이 하나도 없으면 → **OK. 종료.**
- "막아야 함" 이 있고 아직 최대 라운드 전이면 → 다음 라운드로.
- 최대 라운드에 도달했는데도 "막아야 함" 이 남아 있으면 → **종료하되, 통과 못 했다고 솔직히 밝혀라.** OK 인 척하지 마라.

**라운드 N (반복)**
1. 직전 개발자에게 리뷰어의 "막아야 함" 지적을 전달해 계획을 고치게 한다.
2. 리뷰어에게 고친 계획을 다시 넘겨 점검시킨다.
3. 위 **판정** 으로 돌아간다.

## 출력

각 라운드를 짧게 보여줘서 핑퐁이 눈에 보이게 하라:

```
🏓 라운드 1
  개발자: <계획 요약>
  리뷰어: <막아야 함 N개 / 없음>
🏓 라운드 2
  개발자: <무엇을 고쳤는지>
  리뷰어: <...>
✅ 리뷰어 OK (라운드 2에서 통과)
```

그다음 **최종 계획**(마지막으로 승인된 개발자의 계획)을 정리해서 보여줘라.

## 멈추기

마지막에 물어라: **"이 계획대로 진행할까요?"**

- 리뷰어 OK 로 끝났으면: 승인 시 계획대로 구현하고, 다 되면 `/seunghwan-harness:split-commit` 을 제안해도 좋다.
- 최대 라운드까지 OK 를 못 받았으면: 남은 "막아야 함" 이 무엇인지 보여주고, 승환님에게 어떻게 할지 물어라. 승인 전에는 **파일을 고치거나 커밋하지 마라.**
