지금까지 작업한 변경을 **기능 단위로 쪼개서** 여러 개의 커밋으로 만들어줘. 커밋 메시지는 이 저장소 스타일에 맞춰서.

## 순서

1. `git status` 와 `git diff` (staged/unstaged 모두)로 현재 변경 전체를 파악해줘.
2. 변경이 하나도 없으면 그렇다고 알리고 멈춰줘.
3. 변경을 **논리적 기능 단위로 묶어줘.** 예를 들면:
   - 서로 다른 기능은 다른 커밋으로
   - 리팩터링과 새 기능은 섞지 않기
   - 버그 수정은 따로
   한 파일 안의 변경이 여러 기능에 걸치면, 나눌 수 있으면 `git add -p` 개념으로 조각내는 것도 고려해줘 (단, 억지로 쪼개지는 말 것).
4. `git log --format='%B' -3` 으로 저장소 커밋 스타일을 확인해서 맞춰줘.

## 먼저 계획을 보여주고 멈추기

바로 커밋하지 말고, **커밋 계획**부터 보여줘:

```
커밋 1: <type>: <제목>
  - 포함 파일/변경: ...
커밋 2: <type>: <제목>
  - ...
```

각 커밋 제목은 **Conventional Commits** 형식 `<type>: <한글 요약>`, 50자 이내, "왜" 중심.
type 은 `feat`(새 기능) · `fix`(버그) · `hotfix`(긴급 수정) · `refactor`(구조 개선) · `docs`(문서) · `test` · `chore`(잡일) · `style` · `perf` 중 그 커밋 성격에 맞는 것으로. 필요하면 `feat(scope): ...` 처럼 스코프도.

기능 단위로 나누면 커밋마다 type 이 자연스럽게 달라집니다 (예: 새 기능은 `feat`, 딸려온 리팩터는 `refactor`, 문서는 `docs`) — 그게 쪼개는 이유이기도 합니다.

그리고 물어줘: **"이렇게 나눠서 커밋할까요?"**

## 승인 후

승환님이 승인하면, 계획대로 커밋을 하나씩 만들어줘:
- 각 커밋에 해당하는 파일만 `git add` 하고 `git commit`.
- `$ARGUMENTS` 에 지시가 있으면(예: "영어로", "3개로") 반영.
- 커밋 메시지 끝에 이 줄을 붙여줘:
  `Co-Authored-By: Claude Opus 4.8 (1M context) <noreply@anthropic.com>`

**push는 하지 마.** 커밋까지만. push는 승환님이 직접 하거나 따로 시킬 때만.
